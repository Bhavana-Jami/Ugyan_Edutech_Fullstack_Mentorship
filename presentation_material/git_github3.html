What is Git?

Git is a distributed version control system that helps you track changes in your code over time.

üéØ Key Benefits

  • Track Changes: See what changed, when, and who made the changes
  • Collaboration: Multiple developers can work on the same project
  • Backup: Your code is safely stored in multiple places
  • Branching: Work on different features simultaneously

üåü Real-World Example

Imagine you're writing a book with friends. Git is like having a magical notebook that:

  • Remembers every version of every chapter
  • Lets everyone work on different chapters simultaneously
  • Combines everyone's work seamlessly
  • Lets you go back to any previous version

Getting Started

1. Install Git

  • Windows: Download from git-scm.com
  • Mac: brew install git or from git-scm.com
  • Linux: sudo apt install git (Ubuntu/Debian)

2. Configure Git

# Set your name and email (required for commits) git config --global user.name "Your Name" git config --global user.email "your.email@example.com" # Check your configuration git config --list

Making Your First Commit

Let's walk through the process of making your first commit and see how file statuses change.

Check status again git status Changes to be committed: new file: hello.txt # 5. Commit the changes git commit -m "Add hello.txt with greeting message" # 6. Working with Remote Repositories

Now let's learn how to connect your local repository with a remote one and synchronize changes.

Adding a Remote Repository

# Add GitHub as remote origin git remote add origin https://github.com/username/repository.git # View remote repositories git remote -v origin https://github.com/username/repository.git (fetch) origin https://github.com/username/repository.git (push)

Synchronizing Changes

# Push changes to remote repository git push origin main # Pull changes from remote repository git pull origin main # Push and set upstream (first time) git push -u origin main # After this, you can just use: git push

üîÑ Sync Operations

  • Push: Send your local commits to remote repository
  • Pull: Download remote commits to your local repository
  • Fetch: Download remote commits but don't merge them
  • Clone: Create a local copy of a remote repository

‚ö†Ô∏è Common Issue

If you get an error when pushing, you might need to pull first: git pull origin main

Branching Basics

Branches allow you to work on different features or experiments without affecting the main codebase.

# Create a new branch git branch feature-login # Switch to the new branch git checkout feature-login # Create and switch in one command git checkout -b feature-login # List all branches git branch # Switch back to main branch git checkout main # Delete a branch git branch -d feature-login

üåø Branch Naming Conventions

  • feature/ - New features (e.g., feature/user-authentication)
  • bugfix/ - Bug fixes (e.g., bugfix/login-error)
  • hotfix/ - Urgent fixes (e.g., hotfix/security-patch)

Practical Example: Building a Website

Let's create a simple website project and use Git to track our progress.

# 1. Create project and initialize Git mkdir awesome-website cd awesome-website git init # 2. Create initial files echo "<h1>My Awesome Website</h1>" > index.html echo "body { font-family: Arial; }" > styles.css # 3. Add and commit git add . git commit -m "Initial website structure" # 4. Create a new feature branch git checkout -b add-navigation # 5. Add navigation (edit index.html) echo "<nav><a href='#'>Home</a></nav>" >> index.html # 6. Commit the change git add index.html git commit -m "Add navigation menu" # 7. Switch back to main and merge git checkout main git merge add-navigation

Next Steps & Best Practices

üéØ Best Practices

  • Use branches for new features
  • Pull before pushing to avoid conflicts
  • Don't commit sensitive information (passwords, API keys)
  • Use .gitignore to exclude unnecessary files
